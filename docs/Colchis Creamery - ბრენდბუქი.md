PostgreSQL-ის არქიტექტურის სწორად დაგეგმვა ყველაზე კრიტიკული ნაბიჯია, რადგან ის პირდაპირ განსაზღვრავს თქვენი სისტემის სისწრაფეს, მასშტაბირებადობასა და უსაფრთხოებას. რადგან Colchis Creamery-ის პლატფორმა ემსახურება როგორც საცალო მომხმარებლებს, ისე საბითუმო პარტნიორებსა და ადმინისტრატორებს, მონაცემთა ბაზა რელაციურად (Relations) ისე უნდა აეწყოს, რომ მარაგები, ფინანსები და ლოგისტიკა ერთმანეთში არ აირიოს.

აი, თქვენი მონაცემთა ბაზის (Database Schema) ოპტიმიზებული სტრუქტურა და ცხრილებს შორის კავშირები:

### **1\. `Users` (მომხმარებლების ცხრილი)**

აქ ინახება ყველას მონაცემი: თქვენი გუნდის, B2C მყიდველების და B2B პარტნიორების. როლების (Roles) გამიჯვნა აქ უმნიშვნელოვანესია.

* `id` (UUID, Primary Key)  
* `email` (String, Unique)  
* `password_hash` (String) \- *დაშიფრული პაროლი.*  
* `role` (Enum) \- *მნიშვნელობები: `ADMIN`, `B2C_CUSTOMER`, `B2B_PARTNER`.*  
* `company_name` (String, Nullable) \- *ივსება მხოლოდ B2B პარტნიორებისთვის.*  
* `stripe_customer_id` (String, Nullable) \- *Stripe-ის მომხმარებლის უნიკალური ID მომავალი სწრაფი გადახდებისთვის.*  
* `created_at` (Timestamp)

### **2\. `Products` (პროდუქციის და მარაგების ცხრილი)**

ერთიანი კატალოგი, საიდანაც ნაშთი ავტომატურად ჩამოიჭრება, მიუხედავად იმისა, ყველი Amazon-ის ლინკით გაიყიდება (თუ API-თ დააკავშირებთ), თქვენი საიტიდან თუ საბითუმო პორტალიდან.

* `id` (UUID, Primary Key)  
* `sku` (String, Unique) \- *უნიკალური შტრიხკოდი/იდენტიფიკატორი.*  
* `name` (String) \- *მაგ: "Premium Artisanal Sulguni".*  
* `price_b2c` (Decimal) \- *საცალო ფასი.*  
* `price_b2b` (Decimal) \- *საბითუმო ფასი პარტნიორებისთვის.*  
* `stock_quantity` (Integer) \- *საერთო მარაგი Co-packer-ის საწყობში, ოჰაიოში.*  
* `is_active` (Boolean) \- *საიტზე გამოსაჩენად.*

### **3\. `Contracts` (ხელშეკრულებების ცხრილი \- Adobe Sign ინტეგრაცია)**

ეს ცხრილი უკავშირდება `Users` ცხრილს (1-to-1 ან 1-to-Many კავშირით) და ემსახურება ფურცელომანიის სრულად ჩანაცვლებას.

* `id` (UUID, Primary Key)  
* `partner_id` (UUID, Foreign Key \-\> `Users.id`) \- *რომელ პარტნიორს ეკუთვნის.*  
* `adobe_sign_agreement_id` (String) \- *Adobe-დან დაბრუნებული უნიკალური ID.*  
* `status` (Enum) \- *მნიშვნელობები: `PENDING`, `OUT_FOR_SIGNATURE`, `SIGNED`, `EXPIRED`.*  
* `signed_document_url` (String) \- *ხელმოწერილი PDF-ის ლინკი უსაფრთხო სერვერზე (მაგ: AWS S3).*  
* `valid_until` (Date) \- *ხელშეკრულების ვადის ამოწურვის თარიღი.*

### **4\. `Orders` (შეკვეთების მთავარი ცხრილი)**

აქ შემოდის როგორც საცალო, ისე დიდი საბითუმო შეკვეთები Midwest-ის შტატებიდან.

* `id` (UUID, Primary Key)  
* `user_id` (UUID, Foreign Key \-\> `Users.id`) \- *ვინ შეუკვეთა.*  
* `order_type` (Enum) \- *მნიშვნელობები: `B2C`, `B2B`.*  
* `total_amount` (Decimal) \- *ჯამური გადასახდელი თანხა.*  
* `stripe_payment_intent_id` (String, Nullable) \- *Stripe-ის ტრანზაქციის ID.*  
* `payment_status` (Enum) \- *მნიშვნელობები: `UNPAID`, `PAID`, `REFUNDED`.*  
* `order_status` (Enum) \- *მნიშვნელობები: `PROCESSING`, `READY_FOR_PICKUP`, `SHIPPED`, `DELIVERED`.*  
* `created_at` (Timestamp)

### **5\. `Order_Items` (შეკვეთის დეტალების ცხრილი)**

ვინაიდან ერთ შეკვეთაში შეიძლება რამდენიმე სხვადასხვა სახეობის ყველი იყოს, ეს ცხრილი აკავშირებს `Orders` და `Products` ცხრილებს (Many-to-Many კავშირის რეალიზაცია).

* `id` (UUID, Primary Key)  
* `order_id` (UUID, Foreign Key \-\> `Orders.id`)  
* `product_id` (UUID, Foreign Key \-\> `Products.id`)  
* `quantity` (Integer) \- *რაოდენობა.*  
* `unit_price` (Decimal) \- *ფასი ყიდვის მომენტში (რადგან მომავალში პროდუქტის ფასი შეიძლება შეიცვალოს).*

### **6\. `Shipments` (ლოგისტიკისა და თრექინგის ცხრილი)**

B2B პორტალზე პარტნიორისთვის მიწოდების გრაფიკის საჩვენებლად. უკავშირდება `Orders` ცხრილს (1-to-1 კავშირი).

* `id` (UUID, Primary Key)  
* `order_id` (UUID, Foreign Key \-\> `Orders.id`, Unique)  
* `carrier_name` (String) \- *ლოგისტიკური კომპანიის სახელი ან თქვენი შიდა დისტრიბუცია.*  
* `tracking_number` (String, Nullable)  
* `dispatch_date` (Date) \- *როდის გადის ქარხნიდან.*  
* `estimated_delivery_date` (Date) \- *მიწოდების სავარაუდო თარიღი.*  
* `delivery_address` (String) \- *ზუსტი ლოკაცია.*

**როგორ მუშაობს ეს ყველაფერი Backend-ში (Node.js ლოგიკა):** როდესაც მომხმარებელი საიტზე შეკვეთას აკეთებს და Stripe-ის გადახდა წარმატებით სრულდება (Webhook-ის გავლით იღებთ `payment_intent.succeeded` სტატუსს), Node.js სერვერი:

1. პოულობს შესაბამის ჩანაწერს `Orders` ცხრილში და `payment_status`\-ს გადაიყვანს `PAID` რეჟიმში.  
2. შედის `Order_Items` ცხრილში, ნახულობს რა პროდუქტები გაიყიდა.  
3. აკეთებს განახლებას (UPDATE) `Products` ცხრილში და აკლებს გაყიდულ რაოდენობას `stock_quantity`\-ს, რათა მარაგები ყოველთვის ზუსტი იყოს.

თქვენი სრულიად დამოუკიდებელი (Custom) სისტემისთვის Backend-ის API მარშრუტების (Endpoints) სწორად სტრუქტურირება აუცილებელია. ეს არის "ხიდი" თქვენს მონაცემთა ბაზასა (PostgreSQL) და მომხმარებლის ინტერფეისს (Next.js) შორის.

აქ არის Node.js (Express ან NestJS) API Endpoints-ების სრული ჩამონათვალი, დაყოფილი ლოგიკურ მოდულებად:

### **1\. ავტორიზაცია და მომხმარებლები (Auth & Users)**

ეს მოდული მართავს სისტემაში შემოსვლას და როლების (B2C, B2B, Admin) გამიჯვნას (რეკომენდებულია JWT \- JSON Web Tokens ტექნოლოგიის გამოყენება).

* `POST /api/auth/register`  
  * **მიზანი:** ახალი საცალო (B2C) მომხმარებლის რეგისტრაცია.  
* `POST /api/auth/login`  
  * **მიზანი:** სისტემაში შესვლა (აბრუნებს JWT ტოკენს, რომელიც განსაზღვრავს მომხმარებლის როლს).  
* `GET /api/users/me`  
  * **მიზანი:** ავტორიზებული მომხმარებლის ან B2B პარტნიორის პირადი მონაცემების წამოღება (პროფილის გვერდისთვის).  
* `POST /api/users/partners` **(მხოლოდ Admin-ისთვის)**  
  * **მიზანი:** თქვენი გუნდის მიერ ახალი საბითუმო პარტნიორის სისტემაში ხელით დამატება, რის შემდეგაც მას ეგზავნება პაროლის აღდგენის/შექმნის ბმული.

### **2\. კატალოგი და მარაგები (Products & Inventory)**

ამ ენდფოინთებით Next.js წამოიღებს ინფორმაციას საიტზე გამოსაჩენად.

* `GET /api/products`  
  * **მიზანი:** პროდუქტების სიის წამოღება.  
  * *პარამეტრები (Query):* `?type=b2c` (აჩვენებს საცალო ფასებს) ან `?type=b2b` (აჩვენებს საბითუმო ფასებს ავტორიზებული პარტნიორებისთვის).  
* `GET /api/products/:id`  
  * **მიზანი:** კონკრეტული ყველის დეტალური ინფორმაციის (აღწერა, სურათები, მარაგი) წამოღება პროდუქტის შიდა გვერდისთვის.  
* `POST /api/products` **(მხოლოდ Admin-ისთვის)**  
  * **მიზანი:** ახალი პროდუქტის (მაგ: ახალი ტიპის დაძველებული ყველის) დამატება ბაზაში.  
* `PATCH /api/products/:id/stock` **(მხოლოდ Admin-ისთვის)**  
  * **მიზანი:** მარაგების (`stock_quantity`) ხელით კორექტირება საწყობში მიღების ან ჩამოწერის დროს.

### **3\. შეკვეთები (Orders)**

აქ ხდება ყიდვის პროცესის ინიცირება როგორც საცალო, ისე საბითუმო მომხმარებლების მიერ.

* `POST /api/orders/checkout` (B2C-სთვის)  
  * **მიზანი:** კალათის მონაცემების გაგზავნა. სერვერი ითვლის ჯამურ თანხას, ქმნის `Order`\-ს `UNPAID` სტატუსით და უკან აბრუნებს Stripe-ის `client_secret`\-ს (Payment Intent), რითაც Frontend-ზე გადახდის ფანჯარა იხსნება.  
* `POST /api/orders/b2b` (B2B-სთვის)  
  * **მიზანი:** საბითუმო პარტნიორი აგზავნის დიდი მოცულობის შეკვეთას პორტალიდან. ქმნის `Order`\-ს და ავტომატურად უგენერირებს ინვოისს დასაფარად.  
* `GET /api/orders`  
  * **მიზანი:** პარტნიორი ხედავს მხოლოდ თავის შეკვეთების ისტორიას. ადმინი ხედავს ყველა შეკვეთას.

### **4\. გარე სისტემების ავტომატიზაცია (Webhooks \- უმნიშვნელოვანესი ნაწილი)**

ეს არის ღია URL-ები, რომლებსაც გარე სერვისები (Stripe, Adobe) აკითხავენ რაიმე მოვლენის დასადასტურებლად.

* `POST /api/webhooks/stripe`  
  * **როგორ მუშაობს:** როცა მომხმარებელი საიტზე ბარათით იხდის, Stripe ამ URL-ზე აგზავნის დაშიფრულ სიგნალს (მაგ: `payment_intent.succeeded`). თქვენი სერვერი ამოწმებს სიგნალს, პოულობს შესაბამის შეკვეთას, ანიჭებს `PAID` სტატუსს და ავტომატურად აკლებს გაყიდულ რაოდენობას `Products` ცხრილში (მარაგებიდან).  
* `POST /api/webhooks/adobe-sign`  
  * **როგორ მუშაობს:** როდესაც სუპერმარკეტის ქსელი ელექტრონულად მოაწერს ხელს საბითუმო ხელშეკრულებას, Adobe აგზავნის სიგნალს. სერვერი `Contracts` ცხრილში სტატუსს ცვლის `SIGNED`\-ზე და მომხმარებელს ავტომატურად ეხსნება წვდომა B2B პორტალის დახურულ ფუნქციებზე (შეკვეთებზე).

### **5\. კონტრაქტების მართვა (Contracts API)**

* `POST /api/contracts/generate` **(მხოლოდ Admin-ისთვის)**  
  * **მიზანი:** ადმინ პანელიდან აგზავნით ბრძანებას. Backend იღებს პარტნიორის მონაცემებს მონაცემთა ბაზიდან, აერთებს მას Adobe-ის B2B კონტრაქტის შაბლონთან და პარტნიორს ელ-ფოსტაზე უგზავნის ხელმოსაწერად.  
* `GET /api/contracts/partner/:partnerId`  
  * **მიზანი:** B2B პორტალზე კონკრეტული პარტნიორის აქტიური და არქივირებული ხელშეკრულებების სიის გამოტანა.

### **6\. ლოგისტიკა და თრექინგი (Shipments)**

* `POST /api/shipments` **(მხოლოდ Admin-ისთვის)**  
  * **მიზანი:** გადახდილი შეკვეთისთვის მიწოდების გრაფიკის (Estimated Delivery) შექმნა.  
* `PATCH /api/shipments/:id/status` **(მხოლოდ Admin-ისთვის)**  
  * **მიზანი:** ტვირთის სტატუსის განახლება (მაგ: `Processing` \-\> `Dispatched` \-\> `In Transit` \-\> `Delivered`).  
* `GET /api/shipments/tracking/:orderId`  
  * **მიზანი:** B2B პარტნიორი თავისი პორტალიდან ითხოვს კონკრეტული ტვირთის ადგილმდებარეობას და მიწოდების ზუსტ თარიღს.

შესანიშნავი არჩევანია. Stripe Webhook არის თქვენი ელექტრონული კომერციის პლატფორმის "გული". ეს არის ის მომენტი, როდესაც ვირტუალური კალათა რეალურ ბიზნეს ტრანზაქციად იქცევა.

იმისათვის, რომ მარაგების აწევ-დაწევა (Inventory mismatch) არ მოხდეს, აქ უნდა გამოვიყენოთ **Database Transactions (მონაცემთა ბაზის ტრანზაქციები)**. ეს ნიშნავს, რომ ან ყველა ნაბიჯი (გადახდის დადასტურებაც და მარაგების ჩამოჭრაც) წარმატებით სრულდება ერთდროულად, ან შეცდომის შემთხვევაში სისტემა საწყის მდგომარეობას უბრუნდება (Rollback), რათა ბაზა არ აირიოს.

აი, Node.js (Express.js) და PostgreSQL გარემოსთვის დაწერილი Webhook-ის სუფთა და უსაფრთხო სტრუქტურა:

### **Webhook-ის კოდი (Node.js / Express / pg პაკეტი)**

JavaScript

import express from 'express';

import Stripe from 'stripe';

import { Pool } from 'pg';

// ინიციალიზაცია

const stripe \= new Stripe(process.env.STRIPE\_SECRET\_KEY);

const pool \= new Pool({ connectionString: process.env.DATABASE\_URL });

const router \= express.Router();

// კრიტიკული დეტალი: Stripe-ის ხელმოწერის შესამოწმებლად აუცილებელია 'raw' ფორმატი და არა JSON

router.post('/api/webhooks/stripe', express.raw({ type: 'application/json' }), async (req, res) \=\> {

  const sig \= req.headers\['stripe-signature'\];

  let event;

  try {

    // 1\. უსაფრთხოების ვალიდაცია: ნამდვილად Stripe-დან მოვიდა ეს სიგნალი?

    event \= stripe.webhooks.constructEvent(

        req.body, 

        sig, 

        process.env.STRIPE\_WEBHOOK\_SECRET

    );

  } catch (err) {

    console.error(\`⚠️ Webhook-ის ვალიდაციის შეცდომა: ${err.message}\`);

    return res.status(400).send(\`Webhook Error: ${err.message}\`);

  }

  // 2\. ვუსმენთ მხოლოდ წარმატებული გადახდის ივენთს

  if (event.type \=== 'payment\_intent.succeeded') {

    const paymentIntent \= event.data.object;

    

    // Metadata-ში ვიღებთ შეკვეთის ID-ს, რომელიც გადახდის ინიცირებისას გავატანეთ

    const orderId \= paymentIntent.metadata.orderId; 

    // ვიწყებთ მონაცემთა ბაზის ტრანზაქციას

    const client \= await pool.connect();

    

    try {

      await client.query('BEGIN'); // ტრანზაქციის დაწყება

      // ნაბიჯი A: ვაახლებთ Orders ცხრილს (სტატუსი ხდება PAID)

      await client.query(

        \`UPDATE Orders 

         SET payment\_status \= 'PAID', stripe\_payment\_intent\_id \= $1 

         WHERE id \= $2\`,

        \[paymentIntent.id, orderId\]

      );

      // ნაბიჯი B: მოგვაქვს ამ კონკრეტული შეკვეთის პროდუქტები

      const { rows: orderItems } \= await client.query(

        \`SELECT product\_id, quantity FROM Order\_Items WHERE order\_id \= $1\`,

        \[orderId\]

      );

      // ნაბიჯი C: ვამცირებთ მარაგებს Products ცხრილში

      for (const item of orderItems) {

        await client.query(

          \`UPDATE Products 

           SET stock\_quantity \= stock\_quantity \- $1 

           WHERE id \= $2 AND stock\_quantity \>= $1\`, // დაზღვევა მინუსებში წასვლისგან

          \[item.quantity, item.product\_id\]

        );

      }

      await client.query('COMMIT'); // ყველაფერმა წარმატებით ჩაიარა, ვინახავთ ბაზაში

      console.log(\`✅ შეკვეთა ${orderId} წარმატებით დამუშავდა და მარაგები განახლდა.\`);

    } catch (dbError) {

      await client.query('ROLLBACK'); // მოხდა შეცდომა, ვაუქმებთ ცვლილებებს

      console.error('❌ ბაზის ტრანზაქცია ჩაიშალა. პროცესი გაუქმებულია:', dbError);

      

      // აუცილებელია დავაბრუნოთ შეცდომა, რათა Stripe-მა იცოდეს, რომ თავიდან უნდა სცადოს

      return res.status(500).json({ error: 'Internal Server Error during DB Transaction' });

    } finally {

      client.release(); // ვათავისუფლებთ ბაზასთან კავშირს

    }

  }

  // 3\. Stripe-ს ვპასუხობთ 200 OK-ით, რათა იგივე სიგნალი განმეორებით აღარ გამოგვიგზავნოს

  res.json({ received: true });

});

export default router;

### **3 მთავარი პრინციპი, რაც ამ კოდშია დაცული:**

1. **Security (უსაფრთხოება):** stripe.webhooks.constructEvent არის აბსოლუტური აუცილებლობა. მის გარეშე, ნებისმიერ ჰაკერს შეუძლია თქვენს სერვერზე გამოაგზავნოს ყალბი POST მოთხოვნა და სისტემას ეგონოს, რომ თანხა გადახდილია.  
2. **Atomicity (განუყოფლობა):** BEGIN და COMMIT უზრუნველყოფს იმას, რომ თუ კლიენტმა ფული გადაიხადა, მაგრამ ინტერნეტის ან სერვერის ხარვეზის გამო მარაგების ცხრილი ვერ განახლდა, ბაზა არ აირევა და პრობლემის მოგვარება მარტივად შეგეძლებათ (ROLLBACK).  
3. **Idempotency (განმეორებადობისგან დაცვა):** Stripe ითხოვს 200 OK პასუხს სწრაფად. თუ თქვენი სერვერი დროულად არ უპასუხებს, ის ხელახლა გამოაგზავნის ივენთს. კოდის ეს სტრუქტურა იცავს სისტემას იმისგან, რომ ერთი და იგივე შეკვეთაზე მარაგები ორჯერ არ ჩამოიჭრას.

Adobe Sign-ის ინტეგრაცია მართლაც ის ნაწილია, სადაც თქვენი სისტემა ტრადიციული საიტის ჩარჩოებიდან გამოდის და სრულფასოვან B2B პლატფორმად იქცევა. სუპერმარკეტების ქსელებთან და დისტრიბუტორებთან მუშაობისას, კონტრაქტების მომზადების ავტომატიზაცია უდიდეს დროს დაგიზოგავთ.

ამ პროცესის უკან დგას კონცეფცია, რომელსაც Adobe-ში ეწოდება **Library Document (შაბლონი)** და **Merge Fields (მონაცემების ჩასმა)**.

აი, როგორ მუშაობს ეს ლოგიკა თქვენს Node.js Backend-ში:

### **წინაპირობა:**

1. Adobe Sign-ის პანელში წინასწარ ტვირთავთ PDF ხელშეკრულების შაბლონს (მაგ: "Colchis\_Creamery\_B2B\_Agreement").  
2. ამ შაბლონში, იმ ადგილებში, სადაც ცვლადი ინფორმაცია უნდა ჩაჯდეს (სახელი, კომპანია, ფასდაკლების პროცენტი), ქმნით სპეციალურ ველებს (მაგალითად: CompanyName\_Field, Date\_Field).  
3. Adobe გაძლევთ ამ შაბლონის უნიკალურ ID-ს (libraryDocumentId).

### **კოდის სტრუქტურა (Node.js / Express / Axios)**

JavaScript

import express from 'express';

import axios from 'axios';

import { Pool } from 'pg';

const router \= express.Router();

const pool \= new Pool({ connectionString: process.env.DATABASE\_URL });

// Adobe Sign API-ს მთავარი მისამართი და ავტორიზაციის ჰედერი

const ADOBE\_API\_URL \= 'https://api.na1.adobesign.com/api/rest/v6';

const ADOBE\_HEADERS \= {

  'Authorization': \`Bearer ${process.env.ADOBE\_ACCESS\_TOKEN}\`,

  'Content-Type': 'application/json'

};

router.post('/api/contracts/generate', async (req, res) \=\> {

  // ადმინ პანელიდან ვაგზავნით იმ პარტნიორის ID-ს, ვისთანაც ვდებთ ხელშეკრულებას

  const { partnerId, discountTier } \= req.body;

  const client \= await pool.connect();

  try {

    // 1\. მონაცემთა ბაზიდან მოგვაქვს პარტნიორის დეტალები

    const { rows: partnerRows } \= await client.query(

      \`SELECT email, company\_name FROM Users WHERE id \= $1 AND role \= 'B2B\_PARTNER'\`,

      \[partnerId\]

    );

    if (partnerRows.length \=== 0) {

      return res.status(404).json({ error: 'პარტნიორი ვერ მოიძებნა' });

    }

    const partner \= partnerRows\[0\];

    // 2\. ვამზადებთ ინფორმაციას Adobe-სთვის გასაგზავნად

    const agreementPayload \= {

      fileInfos: \[{

        libraryDocumentId: process.env.ADOBE\_B2B\_TEMPLATE\_ID // წინასწარ ატვირთული შაბლონის ID

      }\],

      name: \`Colchis Creamery \- B2B ხელშეკრულება: ${partner.company\_name}\`,

      participantSetsInfo: \[{

        order: 1,

        role: 'SIGNER',

        memberInfos: \[{ email: partner.email }\] // ვის ვუგზავნით ხელმოსაწერად

      }\],

      signatureType: 'ESIGN',

      state: 'IN\_PROCESS', // 'IN\_PROCESS' ნიშნავს, რომ ეგრევე გაიგზავნება

      

      // 3\. უმთავრესი ნაწილი: მონაცემების ავტომატური ჩასმა (Merge Fields)

      mergeFieldInfo: \[

        { fieldName: 'CompanyName\_Field', defaultValue: partner.company\_name },

        { fieldName: 'Discount\_Field', defaultValue: discountTier },

        { fieldName: 'Date\_Field', defaultValue: new Date().toISOString().split('T')\[0\] }

      \]

    };

    // 4\. ვაგზავნით მოთხოვნას Adobe Sign API-სთან

    const adobeResponse \= await axios.post(

      \`${ADOBE\_API\_URL}/agreements\`, 

      agreementPayload, 

      { headers: ADOBE\_HEADERS }

    );

    const agreementId \= adobeResponse.data.id; // Adobe-მ დაგვიბრუნა ხელშეკრულების უნიკალური ID

    // 5\. ვინახავთ ამ ID-ს ჩვენს PostgreSQL ბაზაში

    await client.query(

      \`INSERT INTO Contracts (partner\_id, adobe\_sign\_agreement\_id, status) 

       VALUES ($1, $2, 'OUT\_FOR\_SIGNATURE')\`,

      \[partnerId, agreementId\]

    );

    res.status(200).json({ 

      message: 'ხელშეკრულება წარმატებით დაგენერირდა და გაიგზავნა',

      agreementId: agreementId 

    });

  } catch (error) {

    console.error('❌ Adobe Sign API შეცდომა:', error.response?.data || error.message);

    res.status(500).json({ error: 'კონტრაქტის გენერაცია ვერ მოხერხდა' });

  } finally {

    client.release();

  }

});

export default router;

### **3 კრიტიკული პრინციპი ამ ლოგიკაში:**

* **Merge Fields ანიჭებს სისტემას მოქნილობას:** თქვენ არ გიწევთ ყოველი კლიენტისთვის PDF-ის ხელით ჩასწორება და ატვირთვა. შაბლონი ერთია, ბაზიდან წამოღებული ინფორმაცია კი დინამიურად იცვლება mergeFieldInfo ობიექტის დახმარებით.  
* **მონაცემთა სინქრონიზაცია (Contracts ცხრილი):** Adobe-სგან დაბრუნებული agreementId\-ის შენახვა აუცილებელია. ამის გარეშე, როცა პარტნიორი ხელს მოაწერს დოკუმენტს, სისტემა ვერ მიხვდება, რომელი ხელშეკრულება გააქტიურდა.  
* **ავტომატური გაგზავნა:** სტატუსი IN\_PROCESS Adobe-ს ეუბნება, რომ დოკუმენტი გენერაციისთანავე გაუგზავნოს პარტნიორს ელ-ფოსტაზე, ოფიციალური და უსაფრთხო ბმულით.

ეს არის ზუსტად ის მომენტი, სადაც თქვენი B2B პლატფორმა დამოუკიდებელ, ავტომატიზებულ ეკოსისტემად იქცევა. როდესაც პარტნიორი (მაგალითად, სუპერმარკეტის ქსელის მენეჯერი) ხელს აწერს გაგზავნილ PDF დოკუმენტს, Adobe Sign მყისიერად აგზავნის სიგნალს თქვენს სერვერზე.

Adobe-ს Webhook-ს ერთი სპეციფიკური მოთხოვნა აქვს — ე.წ. **"Handshake" (ხელის ჩამორთმევის) ვალიდაცია**. სერვერმა აუცილებლად უნდა დააბრუნოს კონკრეტული ჰედერი, რათა Adobe-მ დარწმუნებით იცოდეს, რომ ეს URL ნამდვილად თქვენ გეკუთვნით.

აი, როგორ გამოიყურება ამ პროცესის დამჭერი სუფთა და უსაფრთხო კოდი Node.js-ში:

### **Adobe Sign Webhook-ის ლოგიკა (POST /api/webhooks/adobe-sign)**

JavaScript

import express from 'express';

import { Pool } from 'pg';

import axios from 'axios';

const router \= express.Router();

const pool \= new Pool({ connectionString: process.env.DATABASE\_URL });

router.post('/api/webhooks/adobe-sign', async (req, res) \=\> {

  // 1\. Adobe Webhook-ის სავალდებულო "Handshake" ვალიდაცია

  // Adobe აგზავნის თავის Client ID-ს და ითხოვს, რომ პასუხში იგივე დავუბრუნოთ

  const adobeClientId \= req.headers\['x-adobesign-clientid'\];

  if (adobeClientId) {

    res.setHeader('X-AdobeSign-ClientId', adobeClientId);

  }

  const webhookEvent \= req.body;

  // თუ უბრალოდ ვალიდაციის ტესტია (პირველადი რეგისტრაციისას), ვაბრუნებთ 200 OK-ს

  if (\!webhookEvent || \!webhookEvent.event) {

    return res.status(200).send('Webhook registered successfully.');

  }

  // 2\. ვიჭერთ მხოლოდ იმ ივენთს, როცა ხელშეკრულების პროცესი სრულად დასრულდა (ხელმოწერილია)

  if (webhookEvent.event \=== 'AGREEMENT\_WORKFLOW\_COMPLETED' || webhookEvent.event \=== 'AGREEMENT\_ALL\_SIGNERS\_SIGNED') {

    const agreementId \= webhookEvent.agreement.id;

    const client \= await pool.connect();

    try {

      await client.query('BEGIN'); // ვიწყებთ ტრანზაქციას

      // ნაბიჯი A: ვაახლებთ კონტრაქტის სტატუსს მონაცემთა ბაზაში

      const { rows: contractRows } \= await client.query(

        \`UPDATE Contracts 

         SET status \= 'SIGNED' 

         WHERE adobe\_sign\_agreement\_id \= $1 

         RETURNING partner\_id\`,

        \[agreementId\]

      );

      if (contractRows.length \> 0) {

        const partnerId \= contractRows\[0\].partner\_id;

        // ნაბიჯი B: (სურვილისამებრ) შეგიძლიათ Users ცხრილშიც მონიშნოთ პარტნიორი როგორც სრულად აქტიური

        await client.query(

          \`UPDATE Users SET is\_active\_b2b \= true WHERE id \= $1\`,

          \[partnerId\]

        );

        console.log(\`✅ პარტნიორმა (ID: ${partnerId}) ხელშეკრულებას ხელი მოაწერა. B2B პორტალზე წვდომა გაიხსნა.\`);

      }

      await client.query('COMMIT');

      // ნაბიჯი C: (დამატებითი ლოგიკა) საბოლოო, ხელმოწერილი PDF-ის ჩამოტვირთვა და თქვენს სერვერზე შენახვა

      // აქ შეგიძლიათ გაუშვათ ფუნქცია, რომელიც Adobe API-ს დაურეკავს: GET /agreements/{agreementId}/combinedDocument

      // და შეინახავს ფაილს AWS S3-ზე, შემდეგ კი ლინკს განაახლებს Contracts ცხრილის \`signed\_document\_url\` ველში.

    } catch (error) {

      await client.query('ROLLBACK');

      console.error('❌ მონაცემთა ბაზის განახლება ჩაიშალა:', error);

    } finally {

      client.release();

    }

  }

  // 3\. აუცილებელია დავაბრუნოთ 200 OK სტატუსი Handshake-ის ჰედერით

  res.status(200).send('Event processed');

});

export default router;

### **3 მთავარი დეტალი ამ კოდში:**

1. **მყისიერი აქტივაცია:** როგორც კი ხელმოწერა სრულდება (ხშირად ეს სმარტფონიდან ხდება), სტატუსი ბაზაში წამებში იცვლება SIGNED\-ზე.  
2. **როგორ იხსნება B2B პორტალი:** თქვენს Next.js Frontend-ზე (მომხმარებლის მხარეს) გექნებათ ე.წ. Middleware (დამცავი ფენა). როდესაც კლიენტი შედის თავის კაბინეტში, სისტემა ამოწმებს Contracts ცხრილს. თუ სტატუსი არის PENDING ან OUT\_FOR\_SIGNATURE, მას უჩვენებს გვერდს: *"გთხოვთ, მოაწეროთ ხელი ელ-ფოსტაზე გამოგზავნილ დოკუმენტს"*. მაგრამ როგორც კი სტატუსი SIGNED ხდება (ამ Webhook-ის დამსახურებით), Next.js ავტომატურად უხსნის **New Order (საბითუმო შეკვეთის)** და **Delivery Tracking (ლოგისტიკის)** გვერდებს.  
3. **უსაფრთხოება:** ტრანზაქციები (BEGIN და COMMIT) აქაც დაცულია. თუ რაიმე მიზეზით სერვერი გაითიშა სტატუსის განახლების დროს, ბაზა არ დაზიანდება.

ნამდვილად, Frontend-ის არქიტექტურა არის ის "ვიტრინა", რომელსაც Google-ის ბოტები, ChatGPT, Perplexity და რეალური მომხმარებლები პირველები ხედავენ. იმისათვის, რომ Colchis Creamery-ის საიტმა უსწრაფესად იმუშაოს და ულტრა-პრემიუმ ბრენდის შთაბეჭდილება დატოვოს, **Next.js-ის App Router** (React Server Components) უნდა გამოიყენოთ.

ეს მიდგომა საშუალებას მოგცემთ, საიტის დიდი ნაწილი სერვერზევე დაგენერირდეს (Zero JavaScript client-side), რაც ჩატვირთვის დროს მილიწამებამდე შეამცირებს და AI ბოტებს სუფთა HTML-ს მიაწვდის წასაკითხად.

აი, Frontend არქიტექტურის დეტალური გეგმა თქვენი პლატფორმისთვის:

### **1\. რენდერინგის სტრატეგიები (სიჩქარე და ინდექსაცია)**

Next.js-ში შეგიძლიათ თითოეული გვერდისთვის ინდივიდუალურად განსაზღვროთ, როგორ ჩაიტვირთოს. ეს არის მთავარი საიდუმლო სისწრაფისთვის.

* **Static Site Generation (SSG):** გამოიყენეთ საჯარო გვერდებისთვის (მთავარი გვერდი, "ჩვენ შესახებ", კულინარიული ბლოგი/რეცეპტები).  
  * *როგორ მუშაობს:* ეს გვერდები HTML-ის სახით გენერირდება პროექტის ატვირთვისას (Build time) და ინახება CDN-ზე. როცა მომხმარებელი ან ბოტი შედის, ის მომენტალურად იხსნება ბაზასთან ლოდინის გარეშე.  
* **Server-Side Rendering (SSR):** გამოიყენეთ ონლაინ მაღაზიის კატალოგისთვის (/shop) და კონკრეტული პროდუქტის გვერდისთვის.  
  * *როგორ მუშაობს:* რადგან მარაგები და ფასები შეიძლება შეიცვალოს, სერვერი ყოველ მოთხოვნაზე ახალ HTML-ს აგენერირებს უახლესი მონაცემებით. ბოტები ყოველთვის განახლებულ მარაგს ხედავენ.  
* **Client-Side Rendering (CSR) / Client Components:** გამოიყენეთ მხოლოდ ინტერაქტიული ნაწილებისთვის ("use client" დირექტივით).  
  * *მაგალითები:* Stripe-ის გადახდის ფორმა, კალათაში დამატების ღილაკი, B2B პორტალის ცოცხალი თრექინგის რუკა.

### **2\. GEO (AI ძრავებისთვის) და SEO ოპტიმიზაცია**

იმისათვის, რომ AI-მ მომხმარებელს ზუსტად ურჩიოს თქვენი პროდუქტი (მაგალითად, როცა ეკითხებიან "სად ვიყიდო პრემიუმ ყველი ოჰაიოში?"), საიტს სჭირდება სტრუქტურირებული მონაცემები.

* **Metadata API:** Next.js-ის ჩაშენებული ფუნქცია. თითოეულ გვერდს layout.tsx ან page.tsx ფაილში უნდა გაუწეროთ დინამიური მეტა-მონაცემები (Title, Description, OpenGraph ფოტოები სოციალური ქსელებისთვის).  
* **JSON-LD Schema (კრიტიკულია GEO-სთვის):** ეს არის ფარული სკრიპტი, რომელსაც მხოლოდ AI და საძიებო სისტემები კითხულობენ.  
  * layout.tsx\-ში (მთლიანი საიტისთვის) სვამთ **LocalBusiness** Schema-ს. აქ გაიწერება, რომ ხართ ყველის მწარმოებელი, მდებარეობთ აშშ-ში და გაქვთ კონკრეტული სერვისები.  
  * პროდუქტის შიდა გვერდზე (/shop/\[id\]) სვამთ **Product** Schema-ს (ფასი, რეიტინგი, მარაგშია თუ არა).  
* **sitemap.ts და robots.ts:** Next.js საშუალებას გაძლევთ ეს ფაილები დინამიურად დააგენერიროთ. როცა ბაზაში ახალ რეცეპტს ან ყველს დაამატებთ, Sitemap ავტომატურად განახლდება.

### **4\. Core Web Vitals (სისწრაფის ოპტიმიზაციის ხელსაწყოები)**

პრემიუმ ბრენდის საიტი არ უნდა "ჭედავდეს". Next.js გაძლევთ სპეციალურ კომპონენტებს ამის აღმოსაფხვრელად:

* **next/image:** ულტრა-მაღალი ხარისხის ყველის ფოტოები ავტომატურად უნდა დარენდერდეს ამ კომპონენტით. ის სურათებს WebP ფორმატში აკონვერტირებს, არეგულირებს ზომებს ეკრანის მიხედვით და რთავს Lazy Loading-ს (ჩაიტვირთება მხოლოდ მაშინ, როცა მომხმარებელი ჩამოსქროლავს).  
* **next/font:** კლასიკური (Serif) ფონტები, რომლებსაც ბრენდინგისთვის გამოიყენებთ, პირდაპირ სერვერზე უნდა დაჰოსტოთ ამ მოდულით. ეს გამორიცხავს ტექსტის "ხტომას" საიტის ჩატვირთვისას (Layout Shift).

ეს მართლაც ერთ-ერთი ყველაზე მძლავრი ინსტრუმენტია. როდესაც ChatGPT, Gemini ან Google-ის ბოტები თქვენს საიტზე შემოდიან, ისინი დიზაინს არ უყურებენ — ისინი კოდის სტრუქტურას კითხულობენ.

**JSON-LD (JavaScript Object Notation for Linked Data)** არის ენა, რომლითაც AI-ს პირდაპირ ეუბნებით: *"ეს არის პრემიუმ ქართული ყველი, დამზადებულია ოჰაიოში, ღირს კონკრეტული თანხა და გვაქვს მარაგში"*.

### **რატომ არის ეს კოდი იდეალური?**

1. **generateMetadata ფუნქცია:** როდესაც მომხმარებელი ამ ყველის ლინკს Facebook-ზე ან iMessage-ში აზიარებს, ავტომატურად გამოჩნდება პროდუქტის ლამაზი ფოტო და აღწერა. ეს ორგანული მარკეტინგისთვის კრიტიკულია.  
2. **script type="application/ld+json":** ეს არის ზუსტად ის ოქროს სტანდარტი, რომელსაც Generative Engine Optimization (GEO) ითხოვს. როდესაც ChatGPT ან მე (Gemini) ვეძებთ ინფორმაციას ინტერნეტში, ამ სტრუქტურირებული მონაცემების წაკითხვა წამიერია. ჩვენ ვხედავთ price, availability და brand\-ს ყოველგვარი ტექსტის ანალიზის გარეშე.  
3. **priority ატრიბუტი Image კომპონენტზე:** ეს Google-ის PageSpeed Insights-ში (LCP მეტრიკა) უმაღლეს ქულას მოგიტანთ, რადგან მთავარი ფოტო არ ელოდება დანარჩენი საიტის ჩატვირთვას.

**Next.js Server Actions** მართლაც ერთ-ერთი ყველაზე დიდი ტექნოლოგიური ნახტომია. ტრადიციულად, ასეთი ფორმისთვის დაგჭირდებოდათ ცალკე API ენდფოინთის (/api/leads) შექმნა, შემდეგ Frontend-ზე fetch ან axios მოთხოვნის დაწერა და loading სტატუსების ხელით მართვა.

Server Actions ამ ყველაფერს აუქმებს. თქვენ შეგიძლიათ პირდაპირ დაწეროთ სერვერული ფუნქცია და ის პირდაპირ მიაბათ React-ის ფორმას.

აი, როგორ უნდა აეწყოს ეს პროცესი Colchis Creamery-ის საბითუმო (Wholesale) პარტნიორობის ფორმისთვის:

### **რატომ არის ეს მიდგომა საუკეთესო?**

1. **ნაკლები კოდი:** არ გჭირდებათ ხელით JSON.stringify, fetch, headers და სტატუს კოდების (200, 400, 500\) მართვა. ყველაფერი ბუნებრივად, ერთ ნაკადად მუშაობს.  
2. **პროგრესული გაუმჯობესება (Progressive Enhancement):** თეორიულად, ეს ფორმა იმუშავებს მაშინაც კი, თუ მომხმარებლის ბრაუზერში JavaScript გათიშულია ან ნელა იტვირთება, რადგან ის სტანდარტულ HTML \<form\> მოთხოვნას ეფუძნება.  
3. **უსაფრთხოება:** ვინაიდან submitWholesaleLead ფუნქცია მხოლოდ სერვერზე ეშვება, მონაცემთა ბაზის პაროლები (Connection String) აბსოლუტურად დაცულია და არასოდეს ილიქება (Leak) Frontend-ზე.

უსაფრთხოება და წვდომის კონტროლი (Authorization) სისტემის ის ნაწილია, სადაც შეცდომის დაშვების უფლება არ გვაქვს. როდესაც საქმე ეხება საბითუმო ფასებს, პარტნიორების კონტრაქტებსა და თქვენი ბიზნესის შიდა ანალიტიკას, ეს ინფორმაცია საიმედოდ უნდა იყოს დაცული.

**Next.js Middleware** არის საუკეთესო ინსტრუმენტი ამისთვის. ის დგას მომხმარებელსა და სერვერს შორის და მოთხოვნას ამოწმებს *მანამ*, სანამ გვერდი რენდერინგს დაიწყებს. ეს ნიშნავს, რომ არაავტორიზებული პირი ვერც კი დაინახავს B2B პორტალის ან ადმინ პანელის ჩატვირთვის პროცესს — ის მყისიერად გადამისამართდება შესვლის (Login) გვერდზე.

აქ არის ერთი ტექნიკური ნიუანსი: Next.js Middleware ეშვება **Edge Runtime**\-ში, რაც ნიშნავს, რომ სტანდარტული Node.js-ის jsonwebtoken პაკეტი აქ არ მუშაობს. მის ნაცვლად უნდა გამოვიყენოთ jose პაკეტი, რომელიც იდეალურად არის მორგებული Edge გარემოზე.

### **Middleware-ის სრული კოდი (src/middleware.ts)**

ფაილი უნდა შეიქმნას პირდაპირ src (ან მთავარ) დირექტორიაში, app ფოლდერის გვერდით.

import { NextResponse } from 'next/server';

import type { NextRequest } from 'next/server';

import { jwtVerify } from 'jose';

// ვიღებთ საიდუმლო გასაღებს გარემოს ცვლადებიდან (.env)

const SECRET\_KEY \= new TextEncoder().encode(process.env.JWT\_SECRET);

export async function middleware(req: NextRequest) {

  const path \= req.nextUrl.pathname;

  // 1\. განვსაზღვრავთ, რომელ მარშრუტებს ვიცავთ

  const isAdminRoute \= path.startsWith('/admin');

  const isB2bRoute \= path.startsWith('/b2b-portal');

  // თუ მოთხოვნა არც ადმინზეა და არც B2B პორტალზე (მაგ: საჯარო საიტია), თავისუფლად ვატარებთ

  if (\!isAdminRoute && \!isB2bRoute) {

    return NextResponse.next();

  }

  // 2\. ვეძებთ JWT ტოკენს Cookies-ში (რომელიც Login-ის დროს ჩავწერეთ)

  const token \= req.cookies.get('auth\_token')?.value;

  // თუ ტოკენი საერთოდ არ არის, ვაგზავნით ავტორიზაციის გვერდზე

  if (\!token) {

    return NextResponse.redirect(new URL('/login', req.url));

  }

  try {

    // 3\. ტოკენის დეკოდირება და ვალიდაცია Edge გარემოში

    const { payload } \= await jwtVerify(token, SECRET\_KEY);

    const userRole \= payload.role as string;

    // 4\. მკაცრი როლური კონტროლი (Role-Based Access Control \- RBAC)

    

    // ადმინ პანელში შესვლა შეუძლია მხოლოდ 'ADMIN' როლის მქონეს

    if (isAdminRoute && userRole \!== 'ADMIN') {

      // თუ B2B პარტნიორი შეეცადა ადმინში შესვლას, გადავაგდოთ მის პორტალზე

      if (userRole \=== 'B2B\_PARTNER') {

        return NextResponse.redirect(new URL('/b2b-portal/dashboard', req.url));

      }

      return NextResponse.redirect(new URL('/', req.url));

    }

    // B2B პორტალში შესვლა შეუძლია პარტნიორს (და ადმინსაც, ტესტირებისთვის)

    if (isB2bRoute && userRole \!== 'B2B\_PARTNER' && userRole \!== 'ADMIN') {

      return NextResponse.redirect(new URL('/login', req.url));

    }

    // 5\. თუ ყველაფერი რიგზეა, ვამატებთ მომხმარებლის ID-ს ჰედერში

    // ეს დაგვეხმარება Server Components-ში, რომ ხელახლა აღარ გავშიფროთ ტოკენი

    const response \= NextResponse.next();

    response.headers.set('x-user-id', payload.userId as string);

    response.headers.set('x-user-role', userRole);

    return response;

  } catch (error) {

    // თუ ტოკენს ვადა გაუვიდა (Expired) ან გაყალბებულია, ვშლით მას და ვითხოვთ ახალ ავტორიზაციას

    const response \= NextResponse.redirect(new URL('/login', req.url));

    response.cookies.delete('auth\_token');

    return response;

  }

}

### **3 მთავარი დეტალი, რაც ამ არქიტექტურას გამოარჩევს:**

1. **უსაფრთხოების გარანტია (headers ინტეგრაცია):** კოდის მე-13 ხაზზე, ჩვენ არ ველოდებით, რომ Frontend-მა გამოგვიგზავნოს userId. ჩვენ მას პირდაპირ ვკითხულობთ სერვერის ჰედერიდან, რომელიც წინა ეტაპზე დაწერილმა **Middleware**\-მა ჩასვა ვალიდური JWT ტოკენიდან. ეს გამორიცხავს იმას, რომ ერთმა პარტნიორმა URL-ის შეცვლით სხვა პარტნიორის მონაცემები დაინახოს.  
2. **მონაცემთა პარალელური ამოღება (Promise.all):** ნაცვლად იმისა, რომ სერვერმა ჯერ პარტნიორის მონაცემები ეძებოს, მერე დაელოდოს, მერე კონტრაქტი ეძებოს და ბოლოს შეკვეთები, Promise.all სამივე Query-ს PostgreSQL-ისკენ ერთდროულად უშვებს. შედეგად, დაფა 3-ჯერ უფრო სწრაფად იტვირთება.  
3. **JOIN ოპერაცია ლოგისტიკისთვის:** SQL Query-ში გამოიყენება LEFT JOIN Shipments s ON o.id \= s.order\_id. ამის დახმარებით, შეკვეთის ფინანსური ინფორმაციაც და მისი მოწოდების გრაფიკიც ერთი წაკითხვით მოგვაქვს ბაზიდან და მომხმარებელს ზუსტად ვაჩვენებთ, რა ეტაპზეა პროცესი.

### **2\. დინამიური ფორმა და კალკულაცია (Client Component)**

ახლა შევქმნათ კომპონენტი components/BulkOrderClientForm.tsx. აქ გამოვიყენებთ React-ის useState\-ს, რათა მომხმარებლის მიერ შეყვანილი რაოდენობები რეალურ დროში ავსახოთ.

'use client';

import { useState, useMemo } from 'react';

import { useRouter } from 'next/navigation';

// import { submitB2BOrder } from '@/app/actions/orders'; // Server Action ბაზაში შესანახად

interface Product {

  id: string;

  name: string;

  sku: string;

  price\_b2b: string | number; // ბაზიდან წამოსული Numeric ტიპი

  stock\_quantity: number;

}

interface BulkOrderProps {

  products: Product\[\];

  discountPercentage: number;

  partnerId: string;

}

export default function BulkOrderClientForm({ products, discountPercentage, partnerId }: BulkOrderProps) {

  const router \= useRouter();

  const \[loading, setLoading\] \= useState(false);


  // State: რომელ პროდუქტს რა რაოდენობით უკვეთავს 

  // ობიექტის ფორმატი: { "product\_id": quantity }

  const \[orderItems, setOrderItems\] \= useState\<Record\<string, number\>\>({});

  // რაოდენობის შეცვლის ფუნქცია

  const handleQuantityChange \= (productId: string, qty: string) \=\> {

    const parsedQty \= parseInt(qty, 10);

    const validQty \= isNaN(parsedQty) || parsedQty \< 0 ? 0 : parsedQty;

    setOrderItems(prev \=\> ({

      ...prev,

      \[productId\]: validQty

    }));

  };

  // useMemo: ჯამების ავტომატური დათვლა მხოლოდ მაშინ, როცა orderItems იცვლება

  const { subtotal, discountAmount, finalTotal } \= useMemo(() \=\> {

    let sub \= 0;

    products.forEach(p \=\> {

      const qty \= orderItems\[p.id\] || 0;

      sub \+= qty \* Number(p.price\_b2b);

    });

    const disc \= (sub \* discountPercentage) / 100;

    const final \= sub \- disc;

    return { subtotal: sub, discountAmount: disc, finalTotal: final };

  }, \[orderItems, products, discountPercentage\]);

  // ფორმის გაგზავნა

  const handleSubmit \= async (e: React.FormEvent) \=\> {

    e.preventDefault();

    if (finalTotal \=== 0\) return alert('გთხოვთ, აირჩიოთ მინიმუმ ერთი პროდუქტი.');

    

    setLoading(true);

    try {

      // 1\. აქ ვიძახებთ Server Action-ს, რომელსაც გადავცემთ orderItems-ს

      // const response \= await submitB2BOrder(partnerId, orderItems);

      

      // 2\. წარმატების შემთხვევაში გადაგვყავს პარტნიორი ინვოისების/გადახდის გვერდზე

      // router.push(\`/b2b-portal/invoices/${response.invoiceId}\`);

      alert('შეკვეთა წარმატებით შეიქმნა\!');

      router.push('/b2b-portal/dashboard');

    } catch (error) {

      alert('დაფიქსირდა შეცდომა შეკვეთის გაფორმებისას.');

    } finally {

      setLoading(false);

    }

  };

  return (

    \<form onSubmit={handleSubmit} className="bg-white rounded-lg shadow-sm border border-gray-100 overflow-hidden"\>

      \<div className="overflow-x-auto"\>

        \<table className="w-full text-left border-collapse"\>

          \<thead\>

            \<tr className="bg-\[\#FDFBF7\] text-\[\#2C2A29\] text-sm uppercase tracking-wide"\>

              \<th className="p-4 border-b border-gray-100"\>პროდუქტი\</th\>

              \<th className="p-4 border-b border-gray-100"\>SKU\</th\>

              \<th className="p-4 border-b border-gray-100"\>საბითუმო ფასი\</th\>

              \<th className="p-4 border-b border-gray-100 w-32"\>რაოდენობა (ცალი)\</th\>

              \<th className="p-4 border-b border-gray-100 text-right"\>ჯამი\</th\>

            \</tr\>

          \</thead\>

          \<tbody\>

            {products.map((p) \=\> {

              const qty \= orderItems\[p.id\] || 0;

              const rowTotal \= qty \* Number(p.price\_b2b);

              

              return (

                \<tr key={p.id} className="border-b border-gray-50 hover:bg-gray-50 transition"\>

                  \<td className="p-4 font-medium text-\[\#2C2A29\]"\>{p.name}\</td\>

                  \<td className="p-4 text-sm text-gray-500 font-mono"\>{p.sku}\</td\>

                  \<td className="p-4 text-\[\#CBA153\] font-semibold"\>${Number(p.price\_b2b).toFixed(2)}\</td\>

                  \<td className="p-4"\>

                    \<input

                      type="number"

                      min="0"

                      max={p.stock\_quantity}

                      value={qty || ''}

                      onChange={(e) \=\> handleQuantityChange(p.id, e.target.value)}

                      placeholder="0"

                      className="w-full p-2 border border-gray-300 rounded focus:border-\[\#CBA153\] focus:ring-1 focus:ring-\[\#CBA153\] outline-none transition"

                    /\>

                  \</td\>

                  \<td className="p-4 text-right font-medium text-\[\#2C2A29\]"\>

                    ${rowTotal.toFixed(2)}

                  \</td\>

                \</tr\>

              );

            })}

          \</tbody\>

        \</table\>

      \</div\>

      {/\* ჯამების სექცია \*/}

      \<div className="p-6 bg-\[\#FAFAFA\] border-t border-gray-100 flex flex-col md:flex-row justify-between items-end gap-6"\>

        \<div className="text-sm text-gray-500 w-full md:w-auto"\>

          \* შეკვეთის გაფორმების შემდეგ ავტომატურად დაგენერირდება Stripe ინვოისი გადასახდელად (ACH / Credit Card).

        \</div\>

        

        \<div className="w-full md:w-80 space-y-2"\>

          \<div className="flex justify-between text-gray-600"\>

            \<span\>ქვეჯამი:\</span\>

            \<span\>${subtotal.toFixed(2)}\</span\>

          \</div\>

          {discountPercentage \> 0 && (

            \<div className="flex justify-between text-green-600 font-medium border-b border-gray-200 pb-2"\>

              \<span\>ფასდაკლება ({discountPercentage}%):\</span\>

              \<span\>- ${discountAmount.toFixed(2)}\</span\>

            \</div\>

          )}

          \<div className="flex justify-between text-xl font-serif text-\[\#2C2A29\] pt-2"\>

            \<span\>სულ გადასახდელი:\</span\>

            \<span\>${finalTotal.toFixed(2)}\</span\>

          \</div\>

          

          \<button

            type="submit"

            disabled={loading || finalTotal \=== 0}

            className={\`w-full py-3 mt-4 text-white font-medium rounded transition ${

              loading || finalTotal \=== 0 

                ? 'bg-gray-400 cursor-not-allowed' 

                : 'bg-\[\#CBA153\] hover:bg-opacity-90 shadow-md'

            }\`}

          \>

            {loading ? 'მუშავდება...' : 'შეკვეთის დადასტურება'}

          \</button\>

        \</div\>

      \</div\>

    \</form\>

  );

}

3. **User Experience (UX):** ყველაფერი ერთ გვერდზე ხდება ელვის სისწრაფით. პარტნიორს არ უწევს თითოეული ყველის სათითაოდ "კალათაში დამატება". ის ავსებს ცხრილს, ხედავს ფასდაკლებას და აჭერს ერთ ღილაკს.

Stripe-ის ინვოისების (Stripe Invoicing) სისტემა B2B გაყიდვებში ნამდვილად თამაშის წესებს ცვლის. საცალო (B2C) მყიდველებისგან განსხვავებით, სუპერმარკეტები და დისტრიბუტორები იშვიათად იხდიან ადგილზე საკრედიტო ბარათით. მათ სჭირდებათ ოფიციალური საგადასახადო დოკუმენტი და მოქნილობა, რომ თანხა საბანკო გადარიცხვით (ACH) დაფარონ, რაც თქვენთვისაც მომგებიანია, რადგან ACH გადარიცხვებს საკრედიტო ბარათებზე ბევრად დაბალი საკომისიო აქვს.

აი, რა ხდება კულისებში, როდესაც პარტნიორი აჭერს "შეკვეთის დადასტურებას".

ეს არის app/actions/orders.ts ფაილის (Server Action) ლოგიკა, რომელიც ფორმიდან შემოსულ მონაცემებს ამუშავებს, ამოწმებს ბაზას, უკავშირდება Stripe-ს და პარტნიორს ოფიციალურ ინვოისს უგზავნის.

### **Stripe Invoicing-ის Server Action-ის კოდი**

TypeScript

'use server';

import { Pool } from 'pg';

import Stripe from 'stripe';

const pool \= new Pool({ connectionString: process.env.DATABASE\_URL });

const stripe \= new Stripe(process.env.STRIPE\_SECRET\_KEY, { apiVersion: '2023-10-16' });

export async function submitB2BOrder(partnerId: string, orderItems: Record\<string, number\>) {

  const client \= await pool.connect();

  try {

    await client.query('BEGIN'); // ვიწყებთ უსაფრთხო ტრანზაქციას

    // 1\. მომხმარებლის (პარტნიორის) დეტალების წამოღება

    const { rows: userRows } \= await client.query(

      \`SELECT email, company\_name, stripe\_customer\_id FROM Users WHERE id \= $1\`,

      \[partnerId\]

    );

    const partner \= userRows\[0\];

    // 2\. Stripe Customer ID-ის შემოწმება / შექმნა

    let stripeCustomerId \= partner.stripe\_customer\_id;

    if (\!stripeCustomerId) {

      // თუ პარტნიორს ჯერ არ აქვს Stripe პროფილი, ვუქმნით ახალს

      const customer \= await stripe.customers.create({

        email: partner.email,

        name: partner.company\_name,

        description: 'B2B Wholesale Partner',

      });

      stripeCustomerId \= customer.id;

      

      // ვინახავთ ახალ ID-ს ჩვენს ბაზაში მომავალი შეკვეთებისთვის

      await client.query(\`UPDATE Users SET stripe\_customer\_id \= $1 WHERE id \= $2\`, \[stripeCustomerId, partnerId\]);

    }

    // 3\. ხელშეკრულების ფასდაკლების შემოწმება

    const { rows: contractRows } \= await client.query(

      \`SELECT discount\_percentage FROM Contracts WHERE partner\_id \= $1 AND status \= 'SIGNED'\`,

      \[partnerId\]

    );

    const discountPercentage \= contractRows\[0\]?.discount\_percentage || 0;

    // 4\. პროდუქტების ფასების გადამოწმება მონაცემთა ბაზიდან (უსაფრთხოების მთავარი ნაწილი)

    const productIds \= Object.keys(orderItems);

    const { rows: products } \= await client.query(

      \`SELECT id, name, price\_b2b FROM Products WHERE id \= ANY($1::uuid\[\])\`,

      \[productIds\]

    );

    let dbTotalAmount \= 0;

    // 5\. Stripe Invoice-ის კომპონენტების (Invoice Items) შექმნა

    for (const p of products) {

      const quantity \= orderItems\[p.id\];

      if (quantity \> 0) {

        dbTotalAmount \+= quantity \* Number(p.price\_b2b);

        // Stripe-ს ვეუბნებით, რა დაამატოს ინვოისში

        await stripe.invoiceItems.create({

          customer: stripeCustomerId,

          currency: 'usd',

          description: \`${p.name} (Wholesale)\`,

          unit\_amount: Math.round(Number(p.price\_b2b) \* 100), // Stripe თანხას ცენტებში ითვლის

          quantity: quantity,

        });

      }

    }

    // ფასდაკლების დათვლა და საბოლოო ჯამის მიღება

    const discountAmount \= (dbTotalAmount \* discountPercentage) / 100;

    const finalAmount \= dbTotalAmount \- discountAmount;

    // 6\. ინვოისის დაგენერირება Stripe-ში

    const invoice \= await stripe.invoices.create({

      customer: stripeCustomerId,

      collection\_method: 'send\_invoice', // ნიშნავს, რომ პარტნიორს ელ-ფოსტაზე გაეგზავნება

      days\_until\_due: 30, // გადახდის ვადა: 30 დღე (Net 30 terms)

      // ვრთავთ როგორც ბარათით, ისე ACH საბანკო გადარიცხვით გადახდის მეთოდებს

      payment\_settings: {

        payment\_method\_types: \['card', 'us\_bank\_account'\], 

      },

      // თუ ფასდაკლება ეკუთვნის, ვამატებთ როგორც ცალკე კუპონს ან აღწერას

      description: discountPercentage \> 0 ? \`გამოყენებულია ${discountPercentage}% B2B ფასდაკლება\` : '',

    });

    // 7\. ინვოისის ფინალიზაცია და ავტომატური გაგზავნა პარტნიორის ელ-ფოსტაზე

    const finalizedInvoice \= await stripe.invoices.finalizeInvoice(invoice.id);

    await stripe.invoices.sendInvoice(invoice.id);

    // 8\. შეკვეთის შენახვა ჩვენს PostgreSQL ბაზაში

    const { rows: orderRows } \= await client.query(

      \`INSERT INTO Orders (user\_id, order\_type, total\_amount, stripe\_invoice\_id, payment\_status, order\_status) 

       VALUES ($1, 'B2B', $2, $3, 'UNPAID', 'PROCESSING') RETURNING id\`,

      \[partnerId, finalAmount, finalizedInvoice.id\]

    );

    const newOrderId \= orderRows\[0\].id;

    // დეტალების (Order\_Items) შენახვა

    for (const p of products) {

      const quantity \= orderItems\[p.id\];

      if (quantity \> 0) {

        await client.query(

          \`INSERT INTO Order\_Items (order\_id, product\_id, quantity, unit\_price) VALUES ($1, $2, $3, $4)\`,

          \[newOrderId, p.id, quantity, p.price\_b2b\]

        );

      }

    }

    await client.query('COMMIT'); // ყველაფერი წარმატებით შენახულია

    return { 

      success: true, 

      orderId: newOrderId,

      invoiceUrl: finalizedInvoice.hosted\_invoice\_url // ლინკი, სადაც პარტნიორი შეძლებს გადახდას

    };

  } catch (error) {

    await client.query('ROLLBACK');

    console.error('B2B Order submission error:', error);

    throw new Error('შეკვეთის გაფორმება ვერ მოხერხდა.');

  } finally {

    client.release();

  }

}

### **როგორ მუშაობს ეს ლოგიკა რეალურ ცხოვრებაში:**

1. **უსაფრთხოების შემოწმება (ნაბიჯი 4):** Frontend-ზე გამოთვლილ ჯამს საერთოდ არ ვენდობით. სერვერი თავად იღებს ფასებს ბაზიდან, გადაამრავლებს შეკვეთილ რაოდენობაზე და ასე ითვლის ზუსტ თანხას.  
2. **Net 30 პირობები (ნაბიჯი 6):** days\_until\_due: 30 არის ამერიკულ B2B ბაზარზე მიღებული სტანდარტი. ეს პარტნიორს აძლევს 30 დღეს ინვოისის დასაფარად. Stripe ამას ავტომატურად გააკონტროლებს და გადახდის დაგვიანების შემთხვევაში შეხსენების მეილებს თავად გაუგზავნის.  
3. **Hosted Invoice Page (ნაბიჯი 7):** კოდი აბრუნებს hosted\_invoice\_url\-ს. Stripe თავად აგენერირებს ულამაზეს, დაცულ ვებგვერდს, სადაც პარტნიორი ხედავს Colchis Creamery-ის ლოგოს, შეკვეთილ პროდუქტებს და ღილაკს "Pay". იქვე შეუძლია უსაფრთხოდ მიაბას თავისი კომპანიის საბანკო ანგარიში (ACH) ან გადაიხადოს ბარათით.  
4. **რა ხდება გადახდის შემდეგ?:** ზუსტად ის **Stripe Webhook**, რომელიც ჩვენ წინა საუბრებში დავწერეთ, იჭერს მომენტს, როცა ეს ინვოისი დაიფარება (invoice.paid ივენთი), მონაცემთა ბაზაში შეკვეთის სტატუსს PAID\-ზე ცვლის და საწყობის მარაგებიდან ყველის ამ მოცულობას ავტომატურად აკლებს.

პირველ რიგში, ვნახოთ როგორ იწერება ლოგისტიკის განახლების სერვერული ლოგიკა, შემდეგ კი განვიხილოთ სრული ადმინ პანელის სტრუქტურა და ანალიტიკის მოდული.

### **1\. ლოგისტიკის განახლება (Order Fulfillment) \- Server Action**

როდესაც თქვენი გუნდის წევრი ადმინ პანელში შეკვეთას აიღებს და ტვირთს საექსპედიტორო კომპანიას (ან საკუთარ დისტრიბუციას) გადასცემს, ის ავსებს მარტივ ფორმას (Tracking Number, გადამზიდი, მიტანის თარიღი).

აი, კოდი (app/actions/logistics.ts), რომელიც ამ მონაცემებს ბაზაში ინახავს და B2B პორტალზე მომენტალურად აისახება:

TypeScript

'use server';

import { Pool } from 'pg';

import { revalidatePath } from 'next/cache';

const pool \= new Pool({ connectionString: process.env.DATABASE\_URL });

export async function updateOrderShipment(formData: FormData) {

  const orderId \= formData.get('orderId') as string;

  const newStatus \= formData.get('status') as string; // მაგ: 'SHIPPED', 'DELIVERED'

  const carrierName \= formData.get('carrierName') as string;

  const trackingNumber \= formData.get('trackingNumber') as string;

  const deliveryDate \= formData.get('deliveryDate') as string;

  const client \= await pool.connect();

  try {

    await client.query('BEGIN');

    // 1\. ვაახლებთ შეკვეთის მთავარ სტატუსს

    await client.query(

      \`UPDATE Orders SET order\_status \= $1 WHERE id \= $2\`,

      \[newStatus, orderId\]

    );

    // 2\. ვქმნით ან ვაახლებთ ჩანაწერს Shipments ცხრილში (UPSERT ლოგიკა)

    await client.query(

      \`INSERT INTO Shipments (order\_id, carrier\_name, tracking\_number, estimated\_delivery\_date, dispatch\_date) 

       VALUES ($1, $2, $3, $4, NOW())

       ON CONFLICT (order\_id) 

       DO UPDATE SET 

          carrier\_name \= EXCLUDED.carrier\_name,

          tracking\_number \= EXCLUDED.tracking\_number,

          estimated\_delivery\_date \= EXCLUDED.estimated\_delivery\_date\`,

      \[orderId, carrierName, trackingNumber, deliveryDate\]

    );

    await client.query('COMMIT');

    // 3\. უმთავრესი ნაბიჯი: ვაწმენდინებთ Next.js-ს ქეშს პარტნიორის დაფისთვის

    // ეს უზრუნველყოფს იმას, რომ პარტნიორის ეკრანზე ინფორმაცია გვერდის დარეფრეშებისთანავე განახლდეს (Live Update)

    revalidatePath('/b2b-portal/dashboard');

    

    return { success: true };

  } catch (error) {

    await client.query('ROLLBACK');

    console.error('Shipment update error:', error);

    return { error: 'ლოგისტიკის განახლება ვერ მოხერხდა.' };

  } finally {

    client.release();

  }

}

### **2\. ადმინ პანელის სრული არქიტექტურა (რა უნდა იყოს შიგნით?)**

სრულფასოვანი მართვისთვის, ლოგისტიკისა და B2B კონტრაქტების გარდა, თქვენი მორგებული (Custom) ადმინ პანელი შემდეგ მოდულებს უნდა შეიცავდეს:

* **B2C საცალო შეკვეთების მართვა:** ცალკეული სექცია სტანდარტული ონლაინ მაღაზიიდან შემოსული მცირე შეკვეთებისთვის. აქ გამოჩნდება საცალო მყიდველების მისამართები და Stripe-ის ბარათით გადახდის სტატუსები.  
* **გლობალური მარაგები (Global Inventory):** საწყობის ცენტრალური მართვა. როდესაც ოჰაიოს საწარმოდან ყველის ახალ პარტიას მიიღებთ, აქ ხელით დაამატებთ რაოდენობას Products ცხრილში, რათა საიტმა Out of Stock აღარ აჩვენოს.  
* **კონტენტის მართვის სისტემა (CMS & Marketing):** სექცია ბლოგისა და რეცეპტების დასამატებლად. ორგანული ტრაფიკის (SEO) მოსაზიდად, აქ თქვენი გუნდი ატვირთავს სტატიებსა და იდეებს. მაგალითად, შეძლებთ დაამატოთ ინსტრუქციები, თუ როგორ გამოიყენონ მომხმარებლებმა თქვენი პრემიუმ ყველი იდეალური ბურგერებისა და სტეიკ სენდვიჩების მოსამზადებლად.  
* **B2B Lead Management:** ახალი პარტნიორობის მოთხოვნების განხილვა, სადაც ერთი კლიკით გაუგზავნით Adobe Sign-ის ხელშეკრულებებს სუპერმარკეტებს.

### **3\. Ultimate Analytics Dashboard (ანალიტიკა და ექსპორტი)**

მონაცემები არის თქვენი ბიზნესის კომპასი. ადმინ პანელის მთავარ გვერდზე (app/admin/analytics/page.tsx) აუცილებელია ძლიერი რეპორტინგის მოდული, რომელიც გიჩვენებთ სრულ სურათს.

**რას უნდა მოიცავდეს ანალიტიკის გვერდი:**

* **გაყიდვების განაწილება (Sales Split):** ვიზუალური გრაფიკი, რომელიც გაჩვენებთ შემოსავლების პროცენტულობას: რამდენი პროცენტი მოდის B2C საიტიდან (ბარათით გადახდები), რამდენი B2B პარტნიორებიდან (ACH გადარიცხვები) და რამდენი Amazon-დან.  
* **საუკეთესოდ გაყიდვადი პროდუქტები (Top Performers):** რომელი ყველის სახეობა იყიდება ყველაზე სწრაფად, რათა საწარმოო სიმძლავრეები სწორად დაგეგმოთ.  
* **მომხმარებელთა ზრდის დინამიკა (Customer Acquisition):** ახალი საცალო დარეგისტრირებული მომხმარებლებისა და ახალი B2B პარტნიორების რაოდენობა თვეების მიხედვით.

**ინფორმაციის ექსპორტი (CSV / Excel):**

ანალიტიკის გვერდზე აუცილებლად უნდა გქონდეთ **"Export to CSV"** ღილაკი. ეს კრიტიკულია ბუღალტერიისთვის და ფინანსური მოდელირებისთვის.

*როგორ მუშაობს ექსპორტი ტექნიკურად:*

თქვენ ქმნით სპეციალურ API მარშრუტს (მაგ: GET /api/export/orders). როცა ადმინი აჭერს ღილაკს, სერვერი იღებს მონაცემებს PostgreSQL-დან, აკონვერტირებს მას მძიმით გამოყოფილ ტექსტად (CSV ფორმატში papaparse ან json2csv ბიბლიოთეკის დახმარებით) და უბრუნებს ბრაუზერს, როგორც ჩამოსატვირთ ფაილს. ამ ფაილის გახსნას პირდაპირ Excel-ში ან Google Sheets-ში შეძლებთ.

მონაცემთა ექსპორტი (CSV) ბიზნესის ფინანსური მართვისა და ბუღალტერიისთვის უმნიშვნელოვანესი ინსტრუმენტია. ამერიკულ ბაზარზე ოპერირებისას (მით უმეტეს, როცა Stripe-ის ინვოისებსა და საბითუმო პარტნიორებთან გაქვთ საქმე), ბუღალტერს ზუსტი, ცხრილის სახით დალაგებული ინფორმაცია სჭირდება გადასახადების (Taxes) და რეპორტების სწორად შესადგენად.

Next.js App Router-ში ამის გაკეთება ძალიან ელეგანტურად, **Route Handler**\-ის გამოყენებით შეიძლება. კოდი პირდაპირ დაუკავშირდება მონაცემთა ბაზას, შეკრებს ყველა საჭირო ინფორმაციას ერთ ცხრილად და დააბრუნებს ფაილს, რომელსაც ბრაუზერი ავტომატურად ჩამოტვირთავს.

### **მონაცემთა ექსპორტის (CSV) API ლოგიკა**

TypeScript

import { NextResponse } from 'next/server';

import { Pool } from 'pg';

import { headers } from 'next/headers';

const pool \= new Pool({ connectionString: process.env.DATABASE\_URL });

export async function GET(request: Request) {

  // 1\. უსაფრთხოება: ვამოწმებთ, რომ მოთხოვნას ნამდვილად Admin აკეთებს

  // (ეს ჰედერი წინა ეტაპზე დაწერილმა Middleware-მა უნდა გადმოაწოდოს)

  const role \= headers().get('x-user-role');


  if (role \!== 'ADMIN') {

    return NextResponse.json(

      { error: 'წვდომა შეზღუდულია. მხოლოდ ადმინისტრატორს შეუძლია ექსპორტი.' },

      { status: 403 }

    );

  }

  const client \= await pool.connect();

  try {

    // 2\. მონაცემების წამოღება ბაზიდან (JOIN ოპერაციით, რათა მომხმარებლის სახელიც გამოჩნდეს)

    const { rows } \= await client.query(\`

      SELECT 

        o.id AS "Order\_ID",

        TO\_CHAR(o.created\_at, 'YYYY-MM-DD HH24:MI:SS') AS "Date",

        COALESCE(u.company\_name, u.email) AS "Client\_Name",

        o.order\_type AS "Type",

        o.total\_amount AS "Total\_Amount\_USD",

        o.payment\_status AS "Payment\_Status",

        o.order\_status AS "Logistics\_Status",

        o.stripe\_invoice\_id AS "Stripe\_Invoice\_ID"

      FROM Orders o

      LEFT JOIN Users u ON o.user\_id \= u.id

      ORDER BY o.created\_at DESC

    \`);

    // თუ ბაზა ცარიელია, ვაბრუნებთ შესაბამის შეტყობინებას

    if (rows.length \=== 0) {

      return new NextResponse('ამჟამად მონაცემები არ მოიძებნა.', { status: 404 });

    }

    // 3\. JSON მონაცემების CSV ფორმატში კონვერტაცია

    // ვიღებთ ცხრილის სათაურებს (პირველი ობიექტის გასაღებები)

    const headersList \= Object.keys(rows\[0\]).join(',');

    

    // ვუვლით თითოეულ რიგს ციკლით და ვაქცევთ მძიმით გამოყოფილ ტექსტად

    const csvRows \= rows.map(row \=\> 

      Object.values(row).map(value \=\> {

        // ტექსტური მნიშვნელობების გასუფთავება (თუ მძიმე ან ბრჭყალი ურევია ტექსტში, Excel-მა რომ არ აურიოს)

        const stringValue \= value \=== null || value \=== undefined ? '' : String(value);

        return \`"${stringValue.replace(/"/g, '""')}"\`; 

      }).join(',')

    );

    // ვაერთიანებთ სათაურებსა და რიგებს

    const csvContent \= \[headersList, ...csvRows\].join('\\n');

    // 4\. დღევანდელი თარიღის გენერაცია ფაილის სახელისთვის

    const today \= new Date().toISOString().split('T')\[0\];

    const fileName \= \`Colchis\_Creamery\_Sales\_${today}.csv\`;

    // 5\. ბრაუზერისთვის პასუხის დაბრუნება სწორი ჰედერებით, რათა ფაილი ჩამოიტვირთოს

    return new NextResponse(csvContent, {

      status: 200,

      headers: {

        'Content-Type': 'text/csv; charset=utf-8',

        'Content-Disposition': \`attachment; filename="${fileName}"\`,

      },

    });

  } catch (error) {

    console.error('CSV ექსპორტის შეცდომა:', error);

    return NextResponse.json({ error: 'სისტემური ხარვეზი ექსპორტისას.' }, { status: 500 });

  } finally {

    client.release();

  }

}

### **რატომ არის ეს არქიტექტურა იდეალური:**

1. **ავტონომიურობა:** არ იყენებთ მესამე მხარის მძიმე პაკეტებს (როგორიცაა papaparse ან json2csv). სუფთა JavaScript-ით აწყობთ ფორმატს, რაც სერვერზე დატვირთვას ამცირებს.  
2. **მონაცემთა სისუფთავე (COALESCE SQL-ში):** კოდის SELECT ნაწილში გამოყენებულია COALESCE(u.company\_name, u.email). ეს ნიშნავს, რომ თუ B2B პარტნიორია, ცხრილში ჩაწერს კომპანიის სახელს (მაგ: "Kroger"), ხოლო თუ B2C საცალო მყიდველია (ვისაც კომპანიის სახელი არ აქვს), ჩაწერს მის ელ-ფოსტას. ასე ცხრილში ცარიელი (Null) ადგილები არ გექნებათ.  
3. **დროის ზონების კონტროლი:** TO\_CHAR(o.created\_at, 'YYYY-MM-DD HH24:MI:SS') უზრუნველყოფს, რომ ბუღალტერმა ზუსტი, წასაკითხი თარიღები დაინახოს და Excel-მა ფორმატები არ აურიოს.
